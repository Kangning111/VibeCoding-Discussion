# 一些讨论

## 1、AI 规划是否可靠？

> AI 的项目规划有传统软件工程基础，但它默认会倾向于“看起来完整”的方案，容易一上来设计过大。

这就是你用 AI 做 VibeCoding 时最需要控制的地方。

你应该经常要求 AI：

```
不要过度设计
先做最小可运行版本
明确当前阶段不做什么
每一步必须有测试
每个文件说明放在哪个目录
不要写死输入输出
```

尤其对你这种工程计算类项目，最重要的是：

```
先闭环，再智能；
先查表，再 Agent；
先确定数据结构，再接 LangGraph；
先 pytest，再扩展功能。
```

------

**判断标准**

当 AI 给你项目规划时，你可以用这个标准判断它是否靠谱：

```
1. 有没有明确输入输出？
2. 有没有最小可运行闭环？
3. 有没有模块边界？
4. 有没有数据文件和代码分离？
5. 有没有测试用例？
6. 有没有说明当前阶段不做什么？
7. 有没有避免一上来做大而全？
```

如果这些都有，说明它基本符合传统软件工程方法。

如果它一上来就说：

```
使用 LangGraph + FastAPI + React + 数据库 + RAG + 多智能体系统
```

但没有先说明最小闭环，那通常就是“AI 式过度设计”。

## 2、现在编程的几大重难点

### （1）第一重难点：需求定义能力

AI 很擅长写代码，但它不能替你真正理解“你到底要什么”。

> **把模糊目标翻译成清晰、可执行、可测试的软件任务。**

------

### （2）第二重难点：系统拆分能力

AI 可以一次性生成很多代码，但如果你不会拆系统，它生成出来的东西很容易变成一坨。

现在真正重要的是知道项目应该拆成哪些层：

```text
输入层：读取 md / jsonl / metadata
解析层：提取工步信息
决策层：选择表格和参数槽位
查询层：查 jsonl 表
计算层：计算 n、i、vf 等
校核层：检查范围和约束
输出层：生成 markdown / json
测试层：pytest 验证
```

这就是软件工程里的模块化。

AI 可以帮你写每一层，但**人要决定边界在哪里**。

如果边界不清楚，AI 写出来的代码会有几个问题：

```text
一个函数干太多事
数据结构前后不一致
输入输出被写死
逻辑和表格数据混在一起
后面想扩展很困难
```

所以现在编程的第二个重难点是：

> **不是写代码，而是拆代码。**

------

### （3）第三重难点：测试与验证能力

AI 写代码很快，但它写错也很快。

尤其是工程计算类项目，最危险的不是程序报错，而是：

> 程序能跑，但结果是错的。

这些错误语法上都没问题，但工程意义上是错的。

所以现在编程最关键的是：

> **要有能力设计测试用例，判断 AI 写出来的逻辑是否符合业务/工程规律。**

比如你应该要求 AI 写测试：

```text
给定一个粗车外圆工步，应选择表8-1
给定一个精车外圆工步，应选择表8-2
给定 vc 和直径 D，应正确计算 n
给定余量和 ap，应正确计算切削次数 i
更换输入文件路径后，程序仍能运行
更换工步编号后，程序不应写死结果
```

------

### （4）第四重难点：数据结构设计能力

现在很多项目，代码本身不是最难的，难的是数据结构。如果数据结构设计得好，代码会很简单。

例如：

```json
{
  "step_id": "20-01",
  "process_type": "turning",
  "operation_type": "rough_turning",
  "feature": "outer_cylinder",
  "material": "45钢",
  "diameter": 40,
  "ap": 2.0
}
```

这类结构一旦稳定，后面的查表、计算、输出都能围绕它展开。

所以现在编程的一个核心难点是：

> **不是先写函数，而是先定义对象、字段、接口和数据流。**

------

### （5）第五重难点：控制项目复杂度

AI 有一个很明显的问题：它喜欢把项目做大。

你让它写一个小工具，它可能给你：

```text
FastAPI
React
数据库
Docker
LangGraph
多智能体
用户登录
日志系统
配置中心
可视化前端
```

看起来很高级，但对当前阶段没有帮助。

所以 VibeCoding 里的重要能力是：

> **让 AI 在正确的复杂度内工作。**

------

### （6）第六重难点：代码审查能力

**测试负责证明“它在典型场景下对不对”，代码审查负责发现“它为什么可能在别的场景下错”。**

现在不是每一行代码都要你亲手写，但你必须能看懂：

```text
它的输入是什么
输出是什么
有没有写死
有没有异常处理
有没有隐藏假设
有没有单位错误
有没有和需求不一致
测试是否真的覆盖了关键逻辑
```

也就是说，人类程序员的角色正在从：

> 代码生产者

变成：

> 架构师 + 审查者 + 测试者 + 需求定义者。

你不一定要手写所有代码，但你必须能判断 AI 写的代码是否可信。

## 3、数据结构的设计

### （1）最合理的人机分工

| 环节                              | 更适合谁主导 |
| --------------------------------- | ------------ |
| 根据需求提出初版字段              | AI           |
| 把自然语言整理成 JSON / dataclass | AI           |
| 检查字段命名一致性                | AI           |
| 生成 schema / 测试样例            | AI           |
| 判断字段工程含义                  | 人           |
| 判断当前阶段是否过度设计          | 人           |
| 判断字段是否符合教材和工艺逻辑    | 人           |
| 最终确定 schema                   | 人           |

所以不是“AI 能不能做数据结构设计”，而是：

> AI 很适合做数据结构设计的草案生成器和一致性检查器；人负责语义判断和复杂度控制。

------

### （2）你在 VibeCoding 中可以这样要求 AI

以后你让 Codex 或 ChatGPT 帮你设计数据结构时，可以这样写：

```
请只为当前 MVP 设计数据结构，不要过度抽象。

项目目标：
输入工艺规程 md 和指定工步编号，从 metadata 和 jsonl 表中查询切削参数，并输出 markdown 报告。

请输出：
1. 当前阶段必须的数据结构；
2. 每个字段的含义；
3. 字段来源：用户输入 / md解析 / 查表 / 计算 / 默认值；
4. 字段是否必填；
5. 字段单位；
6. 不要引入当前阶段用不到的对象；
7. 最后说明哪些字段是未来扩展时再加入的。
```

这个 prompt 很适合让 AI 参与数据结构设计。

------

### （3）结论

AI擅长的是：

```
归纳结构
生成字段
整理 schema
统一命名
发现缺失
生成样例
```

它不完全擅长的是：

```
判断工程语义
控制项目复杂度
识别真实业务边界
决定当前阶段该不该引入某个抽象
```

所以对你来说，最好的做法是：让 AI 多给几版数据结构方案，你负责判断哪一版最符合“当前 MVP + 后续可扩展”的平衡点。

## 4、测试与验证能力：不要只测代码能不能跑，要测业务逻辑对不对

### （1）先做“手算标准样例”

这是工程计算类项目最重要的方法，也就是你先人工准备 2～3 个非常明确的样例，作为标准答案。

例如：

```
输入：
工步编号：20-01
工步内容：粗车外圆柱面至 φ40
加工类型：粗车
加工对象：外圆
材料：45钢
刀杆尺寸：16×25
ap：≤3
工件直径：40

预期：
应选择表8-1；
ap 是查表条件，不是输出；
输出 f范围；
不得选择表8-2；
不得使用 md 中原有错误切削参数；
```

这个样例就是你的 **golden case，黄金测试样例**。

AI 写代码后，必须通过这个样例。

这类测试不追求数量多，追求“答案确定、逻辑典型、能代表核心业务”。

------

### （2）每一个核心判断都要有测试

你的项目里，测试不要只测最终结果，而要测中间判断。

比如不要只测：

```
最后有没有输出 f、vc、n、i
```

还要分别测：

```
是否正确解析了工步编号
是否正确识别了粗车/精车/半精车
是否正确识别了外圆/端面/内孔
是否正确选择了表8-1或表8-2
是否正确构造 lookup_keys
是否正确把 ap 当作查表条件
是否正确由 vc 和 D 计算 n
是否正确由余量和 ap 计算走刀次数 i
```

这样做的好处是：一旦结果错了，你能知道错在解析、选表、查表还是计算。

------

### （3）测试要覆盖“正常情况 + 边界情况 + 反例”

一个比较好的测试集应该有三类。

**第一类：正常情况**

例如：

```
粗车外圆 → 查表8-1
精车外圆 → 查表8-2
半精车外圆 → 查表8-2
```

**第二类：边界情况**

例如：

```
工件直径刚好等于表中边界值
ap 刚好等于 ≤3 的上限
材料名称是“45钢”，但表中写的是“碳素结构钢和合金结构钢”
工步文本中有多个数字，需要正确识别哪个是直径
```

**第三类：反例**

例如：

```
铣削工步不应该被车削表处理
没有工步编号时应该报清楚错误
没有匹配表格时不应该胡乱返回参数
md 中已有切削参数时不应直接采用
```

反例非常重要，因为 AI 很容易“强行给答案”。

------

### （4）验证输出中必须有“计算过程追踪”

对工程项目来说，最终结果不够，必须有 trace，也就是输出不能只有：

```
推荐 f = 0.4~0.5 mm/r
```

而应该有：

```
识别工步：粗车外圆
选择表格：表8-1
查表条件：
- 加工材料：碳素结构钢和合金结构钢
- 车刀刀杆尺寸：16×25
- 工件直径：40
- ap范围：≤3

查表结果：
- f范围：0.4~0.5 mm/r
```

这类 trace 有两个作用：

第一，方便人审查；第二，方便测试。



## 5、代码审查能力：重点看“隐藏假设”和“数据流”

代码审查建议用一个固定 checklist。

------

### （1）先看输入输出有没有写死

重点检查：

```
文件路径是否写死？
工步编号是否写死？
材料是否写死？
表8-1是否写死在代码逻辑里？
输出文件名是否固定？
测试样例是否污染正式逻辑？
```

也就是说，正式逻辑应该接收参数，而不是在函数内部固定参数。

------

### （2）看数据结构是否前后一致

很多 AI 代码的 bug 不是语法错，而是字段名错。所以你审查时要特别关注：

> 同一个概念是否只有一个字段名？

------

### （3） 看单位有没有明确

工程计算项目里，单位是高危点，要检查代码里有没有明确写出单位换算。

所以字段名尽量写成：

```
vc_m_per_min
diameter_mm
spindle_speed_rpm
feed_mm_per_rev
feed_speed_mm_per_min
```

哪怕你的 JSONL 保留中文原始字段，进入计算模块后也应该转换成带单位的内部字段。

------

### （4） 看 AI 有没有偷换业务语义

这是代码审查里最重要的一点。这类错误只有懂业务的人才能发现。

所以代码审查不是只问：

```
这段代码能不能运行？
```

而是要问：

```
这段代码是否符合表格的真实用途？
这个字段到底是输入、条件、中间量还是输出？
```

------

### （5）看异常处理是否合理

AI 很容易默认所有输入都完美。

但实际使用时会出现：

```
找不到工步编号
材料无法匹配
表格文件不存在
jsonl 某一行字段缺失
工件直径超出表格范围
ap 不在任何范围内
没有 vc，无法计算 n
```

审查时要看代码有没有清楚处理这些情况。

好的程序不一定要全部自动解决，但至少要报清楚：

```
无法匹配表8-1：缺少字段“工件直径”
无法计算主轴转速：缺少 vc_m_per_min
无法找到工步 20-01：请检查 md 文件中的工步编号
```

### 提示词参考

```
请先不要写实现代码。

基于当前项目需求，先为这个 MVP 设计 pytest 测试方案。

项目目标：
输入工艺规程 md 文件路径和工步编号，解析该工步，根据 metadata 选择参数表，从 jsonl 表格中查表，输出切削参数推荐结果和 markdown 计算过程。

约束：
1. 不要写死 md 文件路径；
2. 不要写死工步编号；
3. 不使用 md 中已有的切削参数；
4. metadata 负责选表；
5. jsonl 负责存储表格内容；
6. ap、f、vc、n、i 要区分来源；
7. 表8-1中 ap 是查表条件，不是输出；
8. 表8-1输出 f；
9. n 由 vc 和 D 计算；
10. i 由余量和 ap 计算。

请输出：
1. 单元测试列表；
2. 集成测试列表；
3. 边界测试列表；
4. 反例测试列表；
5. 每个测试对应的测试目的；
6. 建议的 tests/ 目录结构；
7. 暂时不要写业务实现代码。
```

然后再接着说：

```
现在根据上面的测试方案，先实现最小代码，使测试通过。

要求：
1. 每个模块职责单一；
2. 所有核心函数有类型标注；
3. 不要过度设计；
4. 不要引入 LangGraph、数据库、前端；
5. 实现完成后说明每个文件放在哪个目录；
6. 给出运行 pytest 的命令。
```

最后让它自审：

```
请对你刚生成的代码做一次代码审查。

重点检查：
1. 是否写死输入文件、工步编号或表名；
2. 是否存在字段名前后不一致；
3. 是否存在单位错误；
4. 是否错误地把查表条件当成输出；
5. 是否错误使用 md 中已有切削参数；
6. 是否缺少异常处理；
7. 是否存在当前 MVP 不需要的过度设计。

请以表格形式列出问题、影响、修改建议。
```

## 6、VibeCoding核心原则

> **不要依赖 Codex 记住项目；要让项目自己保存项目知识。**

也就是：

```
聊天上下文：只放当前任务
项目文件：保存长期规则、设计决策、数据结构、测试标准
测试用例：约束代码不能跑偏
```

**把测试当成“不会忘的上下文”**

Codex 会忘聊天，但它不会绕过测试。所以你要把所有关键规则都沉淀成测试。这样即使 Codex 后面忘了前面的一些约定，测试也会拦住它。

## 7、AI 时代代码能力需求

你真正需要读懂的是：

```
核心业务规则
关键数据结构
核心算法
输入输出接口
单位换算
边界条件
异常处理
安全相关逻辑
会影响最终结果的判断逻辑
```

应该关注：

```
读接口
读数据结构
读调用链
读测试
读异常处理
读核心业务规则
读代码差异 diff
读 AI 修改了什么
```

特别是 VibeCoding 中，最重要的是读 **diff**。

每次 AI 改代码后，你不要从头读整个项目，而是重点看：

```
它改了哪些文件？
每个文件为什么要改？
有没有改无关文件？
有没有删掉原有逻辑？
有没有引入新的默认值？
有没有让测试变弱？
```

这个比逐行读全项目更有效。
