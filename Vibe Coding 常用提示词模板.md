# Vibe Coding 常用提示词模板

> 适用场景：中大型项目中的 AI 辅助开发 / Vibe Coding。  
> 核心原则：先规划，后实现；先 MVP，后复杂化；先测试锁定，再继续扩展。  
> 推荐用法：把下面的提示词按项目推进顺序复制给 Codex / Cursor / Claude Code / ChatGPT 使用。

---

# 0. 项目启动前：让 AI 帮你梳理项目想法

## Prompt 0-1：项目想法澄清

```markdown
你现在是我的项目规划助手，不要写代码。

我会给你一个项目想法。请你帮我把这个想法整理成适合后续 Vibe Coding 的项目初始说明。

请输出：

1. 项目目标
2. 用户或使用场景
3. 输入
4. 输出
5. 当前阶段应该做什么
6. 当前阶段不应该做什么
7. 最小 MVP 闭环
8. 可能的风险
9. 需要我确认的问题

要求：
1. 不要扩大项目范围。
2. 不要直接设计复杂架构。
3. 不确定的地方写成“待确认假设”。
4. 优先帮助我确定第一阶段 MVP。

项目想法如下：

【粘贴你的项目想法】
```

# 7. 开始执行单个 Task

```text

8. docs/specs/<slice_name>/tasks.md
9. docs/specs/<slice_name>/acceptance_tests.md

现在只实现：

docs/specs/<slice_name>/tasks.md 中的 Task 【编号】。

要求：

1. 只修改 Task 【编号】相关文件。
2. 不要实现其他 task。
3. 不要创建当前 task 用不到的未来模块。
```

# 8. 让 AI 先写测试，再写实现

## Prompt 8-1：测试先行

```text
你现在只作为测试工程师，不要实现功能代码。

请根据当前 spec 为 Task X 设计测试。

要求：
1. 测试应表达通用行为规则，而不是只验证单个固定样例。
2. 至少包含 2~3 个不同输入变体。
3. 至少包含一个异常输入。
4. 不要把测试写得只能通过 hard-code 方式满足。
5. 测试数据尽量使用临时构造的数据，而不是只依赖 examples 中的固定文件。
6. 每个测试都要说明它验证的是哪条业务规则。
7. 如果当前 spec 不足以设计泛化测试，请指出需要补充的规则。
8. 测试名称要清晰表达业务含义。
9. 完成后说明：
   - 新增了哪些测试
   - 每个测试验证什么
   - 为什么这些测试足以约束当前 task

相关 spec 如下：

【粘贴 requirements / acceptance_tests / task 内容】
```

------

## Prompt 8-2：根据测试实现功能

```text
现在请根据刚才新增的测试，实现 Task 【编号】。

要求：

1. 只写通过这些测试所需的最小功能，要求实现通用逻辑，不允许通过判断固定文件名、固定 step_id、固定字符串来通过测试。
2. 不要实现未来功能。
3. 不要修改测试来迁就错误实现。
4. 不要引入不必要依赖。
5. 保持接口简单清晰。
6.如果实现依赖某个格式假设，请在代码注释和文档中说明。
7. 完成后说明运行哪些测试可以验证，如果换一个同类合法输入，为什么该实现仍然有效。
```

# 11. 防止 AI 过度设计

## Prompt 11-1：过度设计检查

```text
请专门检查当前方案是否过度设计。

重点判断：

1. 是否为了未来功能创建了当前用不到的代码文件？
2. 是否引入了不必要的抽象类或接口？
3. 是否引入了当前 MVP 不需要的依赖？
4. 是否把后续切片的逻辑提前实现了？
5. 是否有复杂但当前无法测试的代码？
6. 是否可以用更简单的方案完成当前目标？

请输出：

1. 过度设计点
2. 为什么它们当前不必要
3. 建议删除、延后或简化的内容
4. 简化后的方案
```

# 12. 小范围重构

```text
要求：

3. 不改变测试期望结果。

5. 不修改无关模块。
6. 只解决当前已经出现的结构问题。

请输出：

1. 当前代码存在的问题
2. 重构目标
3. 计划修改的文件
4. 每个文件怎么改
5. 风险点
6. 验证方式
7. 是否需要新增测试

在我确认前不要写代码。
```

# 20. 总结：推荐使用顺序

一个项目从 0 到迭代，推荐按下面顺序使用这些 Prompt：

```text
0-1  项目想法澄清
1-1  判断项目体量
2-1  生成最小文档集
3-1  生成阶段路线图
4-1  模块占位决策
5-1  生成当前切片 spec
6-1  审查任务规划
7-1  执行单个 task
8-1  先写测试
8-2  根据测试实现
9-1  Debug 修复
10-1  代码审查
11-1  过度设计检查
12-1  重构规划
12-2  执行重构
13-1  文档同步
14-1  切片复盘
15-1  规划下一个切片
16-1  记录 ADR
17-1  生成项目交接摘要
18-1  新会话启动
```

最核心的 5 个高频 Prompt 是：

```text
1. 项目启动规划 Prompt
2. 模块占位决策 Prompt
3. 当前切片 spec Prompt
4. 单 task 实现 Prompt
5. 代码审查 / 过度设计检查 Prompt
```
